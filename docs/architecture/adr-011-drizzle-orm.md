# ADR-011: Drizzle ORM for Database Access

## Status

Accepted (2026-03-20)

## Context

POPS currently uses raw SQL with better-sqlite3 for all database access. Service files manually construct parameterized queries and cast results to TypeScript types. Type definitions live separately in `@pops/db-types` as zod schemas. Migrations are hand-written SQL files.

This works for a single app with ~10 tables. With 11 planned app domains and 40+ tables on the roadmap, the approach has scaling problems:

1. **Type safety gap** — SQL queries return `unknown` and are cast to row types. If a migration adds/renames a column but the TypeScript type isn't updated, there's no compile-time error. The schema and types can drift silently.
2. **Boilerplate** — Every CRUD operation is hand-written SQL. The media domain alone has 9 tables and ~30 procedures. Each needs prepare/bind/cast for list, get, create, update, delete. This is repetitive work that a tool should handle.
3. **Manual migrations** — Schema changes require hand-writing ALTER TABLE statements and mentally diffing the current vs desired schema. Easy to miss a column or constraint.

### Options Considered

**A. Stay with raw SQL + better-sqlite3**

Keep the current approach. Accept the boilerplate and type safety gap.

- Pros: No migration cost, full SQL control, zero dependencies beyond better-sqlite3
- Cons: Type drift risk grows with every table. Boilerplate grows linearly with domain count. Migration authoring is manual and error-prone.

**B. Drizzle ORM**

TypeScript-first query builder and schema manager. Schema defined in TypeScript — the schema IS the type. Supports better-sqlite3 as the underlying driver.

- Pros: Type-safe queries (column names checked at compile time), schema-as-code (single source of truth for types and DDL), auto-generated migrations from schema diffs, SQL-like syntax (not a DSL that hides SQL), lightweight (thin layer over the driver, not a heavy runtime like Prisma), first-class SQLite support
- Cons: Migration cost for existing finance services (~8 service files to rewrite), new dependency, team needs to learn the Drizzle API

**C. Kysely**

Type-safe SQL query builder (no schema management).

- Pros: Very close to raw SQL, excellent TypeScript inference, minimal runtime
- Cons: No schema management — still need hand-written migrations. Solves the type safety problem but not the migration problem. Less ecosystem momentum than Drizzle.

**D. Prisma**

Full-featured ORM with schema DSL, migrations, and query client.

- Pros: Mature, well-documented, large community
- Cons: Heavy runtime (query engine binary), its own schema language (not TypeScript), generates a client that abstracts away SQL entirely, poor SQLite support for advanced features, overkill for a single-user self-hosted system

## Decision

**Option B: Drizzle ORM.**

Drizzle hits the sweet spot: type-safe queries and schema management without the weight of a full ORM. The key wins:

1. **Schema is the type.** Define tables in TypeScript. The query builder knows every column name, type, and constraint at compile time. Add a column → it's immediately available in queries. Typo a column name → compile error. No `as MovieRow[]` casts. No separate `@pops/db-types` zod schemas for row shapes.

2. **SQL-like, not SQL-hiding.** Drizzle queries read like SQL. `db.select().from(movies).where(eq(movies.id, id))` is immediately recognisable to anyone who knows SQL. It doesn't invent a new query language or hide joins behind magic methods.

3. **Migration generation.** `drizzle-kit generate` diffs the schema files against the current database and generates SQL migration files. Still reviewable SQL — not opaque binary migrations.

4. **Same driver.** Drizzle uses better-sqlite3 under the hood. Same synchronous API, same performance, same single-file database. We're adding a type layer, not changing the database engine.

### What changes

| Aspect | Before (raw SQL) | After (Drizzle) |
|--------|-------------------|------------------|
| Schema definition | Hand-written SQL in migration files | TypeScript schema files (single source of truth) |
| Types | Separate zod schemas in `@pops/db-types` | Inferred from Drizzle schema (`InferSelectModel<typeof movies>`) |
| Queries | `db.prepare(sql).all(params) as Row[]` | `db.select().from(table).where(condition)` |
| Migrations | Hand-written SQL files | Auto-generated by `drizzle-kit generate`, reviewable SQL |
| `@pops/db-types` | Zod schemas defining row shapes | Re-exports Drizzle inferred types (or deprecated in favour of direct schema imports) |

### What stays the same

- better-sqlite3 as the SQLite driver
- Synchronous database access
- Single SQLite file as the database
- Parameterized queries (Drizzle handles this automatically)
- The migration runner concept (numbered files, tracked in a table)

## Schema file structure

```
apps/pops-api/src/db/
  schema/
    movies.ts           (table definition)
    tv-shows.ts         (table + seasons + episodes)
    watchlist.ts
    watch-history.ts
    comparisons.ts      (dimensions + comparisons + scores)
    transactions.ts     (existing finance)
    entities.ts
    budgets.ts
    ...
    index.ts            (re-export all tables)
  migrations/           (auto-generated SQL files)
  drizzle.config.ts     (drizzle-kit configuration)
  index.ts              (db connection, export drizzle instance)
```

## Example: before and after

**Before (raw SQL):**
```typescript
// service.ts
const db = getDb();
const rows = db
  .prepare('SELECT * FROM movies WHERE id = ?')
  .get(id) as MovieRow | undefined;

// types.ts (separate file)
export const MovieRowSchema = z.object({
  id: z.number(),
  tmdb_id: z.number(),
  title: z.string(),
  // ... 20 more fields manually defined
});
export type MovieRow = z.infer<typeof MovieRowSchema>;
```

**After (Drizzle):**
```typescript
// schema/movies.ts (single source of truth)
export const movies = sqliteTable('movies', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  tmdbId: integer('tmdb_id').unique().notNull(),
  title: text('title').notNull(),
  // ... columns with types, constraints, defaults
});

// service.ts
const row = db.select().from(movies).where(eq(movies.id, id)).get();
// ← row is automatically typed as typeof movies.$inferSelect
// ← eq(movies.id, id) is type-checked: movies.id must exist, id must be number
```

## Consequences

- All new domain modules (media, inventory upgrade, fitness, etc.) use Drizzle from day one
- Existing finance services are migrated to Drizzle as part of Foundation Epic 6
- `@pops/db-types` either re-exports Drizzle inferred types or is deprecated (types come directly from schema imports)
- Migrations are auto-generated but still reviewable SQL — no opaque tooling
- One new dev dependency: `drizzle-orm` + `drizzle-kit` (dev)
- The synchronous nature of better-sqlite3 is preserved — Drizzle's SQLite adapter supports sync mode
- Zod schemas for input validation remain (they validate user input, not database rows) — but row type definitions are replaced by Drizzle inference
