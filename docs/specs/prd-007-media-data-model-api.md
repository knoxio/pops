# PRD-007: Media Data Model & API Module

**Epic:** [00 — Data Model & API Module](../themes/media/epics/00-data-model-api.md)
**Theme:** Media
**Status:** Draft
**ADRs:** [006 — Split Tables](../architecture/adr-006-media-split-tables.md), [008 — Pairwise ELO Ratings](../architecture/adr-008-pairwise-elo-ratings.md), [010 — Universal Object URI](../architecture/adr-010-universal-object-uri.md), [011 — Drizzle ORM](../architecture/adr-011-drizzle-orm.md)

## Problem Statement

The media app needs a data layer before anything else can be built — UI, integrations, tracking, comparisons, recommendations all read from and write to these tables. The current database has no media tables and the API has no media module. This PRD specifies the schema, migration strategy, tRPC router structure, types, and seed data for the media domain.

## Goal

All media tables defined as Drizzle schema files (single source of truth for DDL and TypeScript types). Migrations auto-generated by `drizzle-kit`. A `media/` tRPC module is registered in the API with CRUD procedures for every sub-domain using the Drizzle query builder. Test seed data enables E2E testing from day one.

## Requirements

### R2: Movies Table

**Drizzle schema (`schema/movies.ts`):**

```typescript
import { sqliteTable, integer, text, real, index } from 'drizzle-orm/sqlite-core';
import { sql } from 'drizzle-orm';

export const movies = sqliteTable('movies', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  tmdbId: integer('tmdb_id').unique().notNull(),
  imdbId: text('imdb_id'),
  title: text('title').notNull(),
  originalTitle: text('original_title'),
  overview: text('overview'),
  tagline: text('tagline'),
  releaseDate: text('release_date'),
  runtime: integer('runtime'),
  status: text('status'),
  originalLanguage: text('original_language'),
  budget: integer('budget'),
  revenue: integer('revenue'),
  posterPath: text('poster_path'),
  backdropPath: text('backdrop_path'),
  logoPath: text('logo_path'),
  posterOverridePath: text('poster_override_path'),
  voteAverage: real('vote_average'),
  voteCount: integer('vote_count'),
  genres: text('genres', { mode: 'json' }).notNull().$type<string[]>().default([]),
  createdAt: text('created_at').notNull().default(sql`(datetime('now'))`),
  updatedAt: text('updated_at').notNull().default(sql`(datetime('now'))`),
}, (table) => [
  index('idx_movies_title').on(table.title),
  index('idx_movies_release_date').on(table.releaseDate),
]);
```

- `tmdb_id` is the canonical external identifier (per ADR-007)
- `genres` is a JSON array of genre strings, e.g. `["Action","Sci-Fi"]` — sourced from TMDB, not user-editable
- `poster_path`, `backdrop_path`, `logo_path` store the relative local cache path (per ADR-009)
- `poster_override_path` stores the user-uploaded override path (per ADR-009)
- `imdb_id` stored for cross-referencing but not used as primary identifier
- URI: `pops:media/movie/{id}` (per ADR-010)

### R3: TV Shows, Seasons, Episodes Tables

**Drizzle schema (`schema/tv-shows.ts`):**

```typescript
import { sqliteTable, integer, text, real, index, unique } from 'drizzle-orm/sqlite-core';
import { sql } from 'drizzle-orm';

export const tvShows = sqliteTable('tv_shows', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  tvdbId: integer('tvdb_id').unique().notNull(),
  name: text('name').notNull(),
  originalName: text('original_name'),
  overview: text('overview'),
  firstAirDate: text('first_air_date'),
  lastAirDate: text('last_air_date'),
  status: text('status'),
  originalLanguage: text('original_language'),
  numberOfSeasons: integer('number_of_seasons'),
  numberOfEpisodes: integer('number_of_episodes'),
  episodeRunTime: integer('episode_run_time'),
  posterPath: text('poster_path'),
  backdropPath: text('backdrop_path'),
  logoPath: text('logo_path'),
  posterOverridePath: text('poster_override_path'),
  voteAverage: real('vote_average'),
  voteCount: integer('vote_count'),
  genres: text('genres', { mode: 'json' }).notNull().$type<string[]>().default([]),
  networks: text('networks', { mode: 'json' }).notNull().$type<string[]>().default([]),
  createdAt: text('created_at').notNull().default(sql`(datetime('now'))`),
  updatedAt: text('updated_at').notNull().default(sql`(datetime('now'))`),
}, (table) => [
  index('idx_tv_shows_name').on(table.name),
]);

export const seasons = sqliteTable('seasons', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  tvShowId: integer('tv_show_id').notNull().references(() => tvShows.id, { onDelete: 'cascade' }),
  tvdbId: integer('tvdb_id').unique().notNull(),
  seasonNumber: integer('season_number').notNull(),
  name: text('name'),
  overview: text('overview'),
  posterPath: text('poster_path'),
  airDate: text('air_date'),
  episodeCount: integer('episode_count'),
  createdAt: text('created_at').notNull().default(sql`(datetime('now'))`),
}, (table) => [
  unique().on(table.tvShowId, table.seasonNumber),
  index('idx_seasons_tv_show_id').on(table.tvShowId),
]);

export const episodes = sqliteTable('episodes', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  seasonId: integer('season_id').notNull().references(() => seasons.id, { onDelete: 'cascade' }),
  tvdbId: integer('tvdb_id').unique().notNull(),
  episodeNumber: integer('episode_number').notNull(),
  name: text('name'),
  overview: text('overview'),
  airDate: text('air_date'),
  stillPath: text('still_path'),
  voteAverage: real('vote_average'),
  runtime: integer('runtime'),
  createdAt: text('created_at').notNull().default(sql`(datetime('now'))`),
}, (table) => [
  unique().on(table.seasonId, table.episodeNumber),
  index('idx_episodes_season_id').on(table.seasonId),
]);
```

- `tvdb_id` is the canonical external identifier for TV content (per ADR-007)
- `ON DELETE CASCADE` enforces hierarchy: deleting a show removes all seasons and episodes
- `UNIQUE(tv_show_id, season_number)` prevents duplicate seasons per show
- `UNIQUE(season_id, episode_number)` prevents duplicate episodes per season
- `networks` is a JSON array of network name strings
- URI: `pops:media/tv-show/{id}`, `pops:media/season/{id}`, `pops:media/episode/{id}`

### R4: Watchlist Table

**Drizzle schema (`schema/watchlist.ts`):**

```typescript
import { sqliteTable, integer, text, index, unique, check } from 'drizzle-orm/sqlite-core';
import { sql } from 'drizzle-orm';

export const watchlist = sqliteTable('watchlist', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  mediaType: text('media_type', { enum: ['movie', 'tv_show'] }).notNull(),
  mediaId: integer('media_id').notNull(),
  priority: integer('priority').notNull().default(0),
  notes: text('notes'),
  addedAt: text('added_at').notNull().default(sql`(datetime('now'))`),
}, (table) => [
  unique().on(table.mediaType, table.mediaId),
  index('idx_watchlist_media').on(table.mediaType, table.mediaId),
  index('idx_watchlist_priority').on(table.priority),
]);
```

- Polymorphic reference: `media_type` + `media_id` points to either `movies.id` or `tv_shows.id`
- No database-level FK enforcement for polymorphic references (per ADR-006) — validated in application layer
- `priority` is a sortable integer — lower values appear first. Default 0.
- `UNIQUE(media_type, media_id)` prevents adding the same item twice

### R5: Watch History Table

**Drizzle schema (`schema/watch-history.ts`):**

```typescript
import { sqliteTable, integer, text, index } from 'drizzle-orm/sqlite-core';
import { sql } from 'drizzle-orm';

export const watchHistory = sqliteTable('watch_history', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  mediaType: text('media_type', { enum: ['movie', 'episode'] }).notNull(),
  mediaId: integer('media_id').notNull(),
  watchedAt: text('watched_at').notNull().default(sql`(datetime('now'))`),
  completed: integer('completed').notNull().default(1),
}, (table) => [
  index('idx_watch_history_media').on(table.mediaType, table.mediaId),
  index('idx_watch_history_watched_at').on(table.watchedAt),
]);
```

- For movies: `media_type = 'movie'`, `media_id` → `movies.id`
- For TV: `media_type = 'episode'`, `media_id` → `episodes.id` (episode-level tracking per resolved decision)
- Multiple entries per item allowed (re-watches)
- `completed` flag distinguishes finished watches from abandoned ones

### R6: Comparison Tables

**Drizzle schema (`schema/comparisons.ts`):**

```typescript
import { sqliteTable, integer, text, real, index, unique } from 'drizzle-orm/sqlite-core';
import { sql } from 'drizzle-orm';
import { movies } from './movies.js';

export const comparisonDimensions = sqliteTable('comparison_dimensions', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  name: text('name').unique().notNull(),
  description: text('description'),
  active: integer('active').notNull().default(1),
  sortOrder: integer('sort_order').notNull().default(0),
  createdAt: text('created_at').notNull().default(sql`(datetime('now'))`),
});

export const comparisons = sqliteTable('comparisons', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  dimensionId: integer('dimension_id').notNull().references(() => comparisonDimensions.id),
  movieAId: integer('movie_a_id').notNull().references(() => movies.id),
  movieBId: integer('movie_b_id').notNull().references(() => movies.id),
  winnerId: integer('winner_id').notNull().references(() => movies.id),
  comparedAt: text('compared_at').notNull().default(sql`(datetime('now'))`),
}, (table) => [
  index('idx_comparisons_dimension').on(table.dimensionId),
  index('idx_comparisons_movies').on(table.movieAId, table.movieBId),
  // CHECK(movie_a_id != movie_b_id) and CHECK(winner_id IN (movie_a_id, movie_b_id))
  // enforced via application-level validation in tRPC procedure (Drizzle doesn't support
  // cross-column CHECK constraints declaratively — the generated migration SQL can be
  // manually amended to include them, or rely on the zod refinement in R10)
]);

export const mediaScores = sqliteTable('media_scores', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  mediaType: text('media_type', { enum: ['movie', 'tv_show'] }).notNull(),
  mediaId: integer('media_id').notNull(),
  dimensionId: integer('dimension_id').notNull().references(() => comparisonDimensions.id),
  score: real('score').notNull().default(1500.0),
  comparisonCount: integer('comparison_count').notNull().default(0),
  updatedAt: text('updated_at').notNull().default(sql`(datetime('now'))`),
}, (table) => [
  unique().on(table.mediaType, table.mediaId, table.dimensionId),
  index('idx_media_scores_lookup').on(table.mediaType, table.mediaId),
  index('idx_media_scores_dimension').on(table.dimensionId, table.score),
]);
```

**Key change from epic schema:** The `comparisons` table uses direct FK references to `movies` instead of polymorphic `media_a_type + media_a_id`. Since comparisons are movies-only in v1 (per ADR-008), proper FK constraints are possible and preferred. When TV show comparisons are added later, either:
- Add `tv_comparisons` as a separate table, or
- Migrate to the polymorphic pattern at that time

The `media_scores` table retains the polymorphic pattern because it will eventually store scores for TV shows too, and scores are a read-heavy derived table where FK enforcement matters less.

**CHECK constraints:**
- `movie_a_id != movie_b_id` — can't compare a movie to itself
- `winner_id IN (movie_a_id, movie_b_id)` — winner must be one of the compared pair

  dimensionId: z.number().int().positive(),
  movieAId: z.number().int().positive(),
  movieBId: z.number().int().positive(),
  winnerId: z.number().int().positive(),
}).refine(
  (d) => d.winnerId === d.movieAId || d.winnerId === d.movieBId,
  { message: 'Winner must be one of the compared movies' }

  { message: 'Cannot compare a movie to itself' }
);

### R11: ELO Calculation

Pure function in `comparisons/elo.ts`:

```typescript
interface EloResult {
  newScoreA: number;
  newScoreB: number;
}

function calculateElo(
  scoreA: number,
  scoreB: number,
  winner: 'a' | 'b',
  kFactor: number = 32
): EloResult;
```

- Standard ELO formula: `expectedScore = 1 / (1 + 10^((opponentRating - rating) / 400))`
- K-factor of 32 (responsive to new data, standard for establishing ratings)
- Symmetric: winner gains what loser loses
- Starting score: 1500 per item per dimension

The `submit` procedure:
1. Validate input (both movies exist, both are watched, dimension exists and is active)
2. Get or create `media_scores` rows for both movies on this dimension
3. Calculate new ELO scores
4. Update scores and increment comparison_count
5. Insert `comparisons` row
6. All in a single transaction

**Comparison pair selection (Epic 4):**
```typescript
// Random pair of watched movies for a dimension
const watchedMovieIds = db.selectDistinct({ id: watchHistory.mediaId })
  .from(watchHistory)
  .where(eq(watchHistory.mediaType, 'movie'));

db.select().from(movies)
  .where(inArray(movies.id, watchedMovieIds))
  .orderBy(sql`RANDOM()`)
  .limit(2);
```

**Rankings (Epic 4):**
```typescript
// Top movies for a dimension
db.select({
  ...getTableColumns(mediaScores),
  title: movies.title,
  posterPath: movies.posterPath,
  releaseDate: movies.releaseDate,
  genres: movies.genres,
})
.from(mediaScores)
.innerJoin(movies, and(
  eq(mediaScores.mediaId, movies.id),
  eq(mediaScores.mediaType, 'movie'),
))
.where(eq(mediaScores.dimensionId, dimensionId))
.orderBy(desc(mediaScores.score))
.limit(limit)
.offset(offset);
```

Note: Some complex queries (e.g., TV show episode progress with correlated subqueries) may use Drizzle's `sql` template literal for raw SQL fragments. This is fine — Drizzle supports mixing typed queries with raw SQL where the query builder is insufficient.

## Edge Cases & Decisions

**Q: Should `genres` be a junction table instead of JSON?**
A: No. Genres are TMDB/TheTVDB-sourced reference data, not user-managed. The set of genres is fixed and small (~20). JSON storage supports filtering via `json_each()` and avoids the overhead of a junction table + genre lookup table for data that will never be independently queried. If genre-based analytics become important, extract to a junction table later.

**Q: Should `comparisons` use the polymorphic pattern for future TV support?**
A: No. Movies-only in v1 means we can use proper FK constraints (`movie_a_id REFERENCES movies(id)`). This is better than polymorphic references where we can have them. When TV comparisons are designed, either add a `tv_comparisons` table or migrate to the polymorphic pattern — both are straightforward.

**Q: How is `updated_at` maintained?**
A: Application-level — the service layer sets `updatedAt` via Drizzle's `.set({ updatedAt: sql\`datetime('now')\` })` on every update. No database trigger.

**Q: What about the `initializeSchema` function for fresh databases?**
A: With Drizzle, `drizzle-kit push` or `drizzle-kit migrate` handles both fresh and existing databases. The Drizzle schema files are the single source of truth — no separate `initializeSchema` function needed.

**Q: What about CHECK constraints Drizzle can't express?**
A: Drizzle doesn't support cross-column CHECK constraints declaratively (e.g., `CHECK(movie_a_id != movie_b_id)`). Two options: (a) manually amend the generated migration SQL to add the CHECK, or (b) enforce via application-level validation in the zod refinement (R10). We use both — the zod schema catches it before the query runs, and the CHECK constraint is a safety net in the generated SQL.
