## 0. Framing — this is _finishing_, not green-field

The runtime-discovery / self-registration north-star (MEMORY `project_pillar_mqtt_discovery.md`) is **~70% already built on `main`**. The DB-backed registry, self-registration (TS SDK + Rust parity), live heartbeat status, the shell's external-UI loader, and disk-discovery manifest validation all exist. This phase **finishes the migration off the residual build-time static artifacts that currently run in parallel with the live registry.** Budget accordingly: do **not** scope green-field effort for already-merged infrastructure. Risk concentrates in exactly two tasks — **RD-3** (flip shell boot source to the live snapshot) and **RD-9** (`KnownPillarId`→`string`) — both gated behind seams that already exist.

## 1. Dependency on the relocation + the master seams

**Hard precondition:** all relocation tasks merged. Because every coupling here references **package names** (`@pops/app-ai`, `@pops/overlay-ego`, `@pops/types`), not paths, the `git mv` relocations need **zero edits** to the files in this phase as long as `pnpm-workspace.yaml` globs resolve the names (`G0`/`G1`). `bundle-map.tsx` is the cleanest seam in the whole migration — exploit it, do not touch it during relocation.

**The four seams that make this phase parallelizable** (already in tree):

| Seam                                                     | Where                                           | Role                                                                                                                                |
| -------------------------------------------------------- | ----------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `RegistryEntry` / `defaultRegistryEntries()`             | `pillars/shell/src/app/installed-modules.ts`    | **master seam.** Shell's pillar-agnostic boot input. Swap its feeder `MODULES`→`/pillars`; `walkRegistry()` signature is invariant. |
| `discoverManifestSources()`                              | `libs/module-registry/scripts/known-modules.ts` | disk-walk discovery; decouples the validator from named pillars. Reuse; never re-import pillars.                                    |
| `/core.registry.list` snapshot + SDK `registerWithRetry` | core + `libs/sdk`                               | runtime truth; new pillars discovered for free; the transport MQTT swap lives behind this contract (separate epic).                 |
| `external-ui.tsx` `RemoteUiDescriptor`                   | `pillars/shell/src/app/external-ui.tsx`         | the **only** sanctioned growth path for out-of-repo pillar UIs.                                                                     |

## 2. Why `generated.ts` cannot just be deleted — split role A from role B

`generated.ts` conflates two roles. The decoupling is **splitting them**, not deleting.

| Role                                             | What                                                                                                                                                                                               | Verdict                                                                                                                                                                                                                                                                                                   |
| ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **A — build-time validation manifest**           | cross-pillar invariants computed over all in-repo pillars at build: duplicate ids, dangling `dependsOn`, URI-handler-type collisions, AI-tool-name collisions (`scripts/lib.ts:validateManifests`) | **stays build-time.** This is `module-registry`'s legitimate, contract-respecting job — it reads each pillar's _published_ `./manifest` export, never reaches behind a contract. Passes the extraction litmus (a generic manifest-validator that could live in its own repo and walk any `pillars/` dir). |
| **B — shell boot install-set + settings source** | `MODULES`/`INSTALLED_MODULES` feed shell `defaultRegistryEntries()` and the API settings aggregation                                                                                               | **moves to runtime** `/pillars` snapshot (RD-3 / RD-4).                                                                                                                                                                                                                                                   |

## 6. Isolation interplay (ISO-\* sequencing)

- ISO-R1 (`lib-no-pillar-import`) is `error` for **new** code immediately; the existing module-registry edges are grandfathered via the dep-cruiser **baseline** (`.dependency-cruiser-known-violations.json`) only until RD-1 removes the devDeps and RD-3/RD-4 remove runtime readers.
- After RD-1+RD-3+RD-4, **delete the module-registry baseline entry**; `baseline-guard.sh` (EX-3) then forbids re-adding it (baseline may only shrink).
- The `@pops/<pillar>` list inside ISO-R1's `to.path` is **generated from disk** (same discovery the pillar matrices use) — never goes stale the way `POPS_PILLARS` did, closing the audit's "static-list rot" at the lint layer too.

## 8. MQTT / transport north-star tie-in

The "MQTT discovery" north-star maps onto the **existing** register/heartbeat transport (`libs/sdk/bootstrap`, contacts `registry/transport.rs`). Today: HTTP POST register + heartbeat to core. Swapping the transport to MQTT is a `libs/sdk` + `core` internal change **behind the same `/pillars` snapshot contract** the shell reads after RD-3 — therefore **fully decoupled from this phase**, a separate later epic. This phase only guarantees (a) `pillar-sdk` lands in `libs/sdk` cleanly and (b) the `RegistryEntry`/snapshot contract is the shell's sole input (RD-3 delivers both).

## 9. Correction to the original brief framing

The brief treats registry decoupling as the "deepest changes to build." It is mostly **finishing**, not building: runtime registry, self-registration (TS + Rust), external-UI loader, and disk-discovery validation are already merged. Real risk is **two** tasks — **RD-3** (flip shell boot source) and **RD-9** (`KnownPillarId`→`string`) — both behind seams already in place. Everything else is cleanup of parallel-running static artifacts.
